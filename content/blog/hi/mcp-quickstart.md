आप पहले से एडिटर में हैं। ऐप का ढाँचा तैयार है, पहला डिप्लॉय किसी प्लेटफ़ॉर्म सबडोमेन पर जा चुका है और लोगों को उस पर भेजने से पहले केवल एक असली डोमेन बचा है। यह उसी ऐप को बनाने वाले [कोडिंग एजेंट](/hi/glossary/ai-agent/) सेशन से बाहर निकले बिना रजिस्ट्रेशन करने का क्विकस्टार्ट है: Claude Code, Cursor और Windsurf के लिए सटीक [MCP](https://modelcontextprotocol.io) कनेक्शन कॉन्फ़िगरेशन, पाँच चरणों वाला संक्षिप्त प्रवाह और—वह हिस्सा जिसे अधिकांश डोमेन गाइड छोड़ देते हैं—अभी रजिस्टर किए डोमेन को अभी जारी किए गए डिप्लॉयमेंट की ओर वास्तव में कैसे पॉइंट करें। इसके लिए ब्राउज़र टैब खोलने, चेकआउट फ़ॉर्म भरने या एडिटर छोड़ने की ज़रूरत नहीं है।

इस गाइड में जानबूझकर तीन एडिटर शामिल हैं। यदि आप इसकी जगह OpenAI Codex, Gemini CLI या Claude Desktop इस्तेमाल करते हैं, तो [Namefi पर अपने AI एजेंट से डोमेन कैसे रजिस्टर करें](/hi/blog/ai-agent-register/) में सभी छह क्लाइंट के सत्यापित सेटअप और MCP-नेटिव न होने वाली किसी भी चीज़ के लिए सीधा REST तरीका मिलता है। यहाँ हर चीज़ उसी [Namefi](https://namefi.io) MCP सर्वर से जुड़ती है जिसका दस्तावेज़ीकरण उस मुख्य गाइड में है, इसलिए नीचे की कोई बात उसके विरोध में नहीं है—यह पृष्ठ उसका संक्षिप्त, डेवलपर-टूल-केंद्रित रूप है, जिसमें वह डिप्लॉयमेंट चरण भी है जिसे मुख्य गाइड शामिल नहीं करती।

## डोमेन को एडिटर के भीतर ही क्यों रजिस्टर करें

"जाकर डोमेन रजिस्टर करें" पाँच मिनट के काम के लिए असामान्य रूप से महँगा कॉन्टेक्स्ट बदलाव है: एडिटर छोड़ें, किसी रजिस्ट्रार की साइट खोलें, नाम खोजें, अनचाही गोपनीयता सुरक्षा और ईमेल होस्टिंग बेचने वाली प्रक्रिया से गुज़रें, भुगतान करें, फिर वापस आकर पता लगाएँ कि कौन-से DNS रिकॉर्ड जोड़ने हैं।

विकल्प यह है कि प्रोजेक्ट का ढाँचा बनाने और डिप्लॉय जोड़ने वाला वही एजेंट अंतिम चरण भी सँभाले: नाम जाँचे, उसे रजिस्टर करे और DNS जोड़े—ये सभी काम उसी बातचीत में टूल कॉल के रूप में हों जो आप पहले से कर रहे हैं। [Cloudflare अपने Registrar API के लिए इसी विचार के एक रूप का प्रचार करता है](https://webhosting.today/2026/04/22/ai-agents-can-now-register-domains-no-human-required/#:~:text=An%20agent%20using%20the%20API%20can%20suggest%20domain%20names%2C%20check%20registrability%2C%20and%20complete%20the%20purchase%20without%20the%20user%20leaving%20their%20current%20context)—यह इस बात का प्रमाण है कि यह कोई सीमित पसंद नहीं, बल्कि ऐसा वर्कफ़्लो है जिसकी ओर एक से अधिक रजिस्ट्रार बढ़ रहे हैं। अंत के पास वाला तुलना सेक्शन विशेष रूप से Cloudflare के पहलू को शामिल करता है; Namefi का तरीका [टोकनाइज़्ड डोमेन](/hi/glossary/tokenized-domain/) विकल्प और बिना किसी अकाउंट के वॉलेट से हस्ताक्षरित भुगतान का तरीका जोड़ता है, जिसका विवरण [क्रिप्टो वॉलेट से डोमेन के लिए भुगतान करें](/hi/blog/wallet-checkout/) में है।

## कनेक्शन सेट करें: तीन एडिटर, तीन कॉन्फ़िगरेशन फ़ाइलें

नीचे दिए तीनों एडिटर Streamable HTTP के ज़रिए एक ही एंडपॉइंट, `https://api.namefi.io/mcp`, से जुड़ते हैं और आपकी Namefi [API कुंजी](https://namefi.io/api-key) को `x-api-key` हेडर के रूप में भेजते हैं। हर एडिटर में केवल फ़ाइल फ़ॉर्मेट और उसे लिखने वाला कमांड बदलता है।

### Claude Code

Claude Code के अपने दस्तावेज़ कस्टम हेडर वाले रिमोट HTTP सर्वर को जोड़ने के लिए सीधा CLI कमांड देते हैं:

```bash
claude mcp add --transport http namefi https://api.namefi.io/mcp --header "x-api-key: YOUR_KEY"
```

इसे अपने प्रोजेक्ट के टर्मिनल में एक बार चलाएँ और अपनी असली कुंजी डालें। डिफ़ॉल्ट रूप से यह सर्वर को **local** स्कोप में लिखता है—यह केवल आपको, केवल इसी प्रोजेक्ट में उपलब्ध होगा। इसके बजाय इसे मशीन के हर प्रोजेक्ट में उपलब्ध कराने के लिए `--scope user` जोड़ें और `claude mcp list` से पुष्टि करें कि कनेक्शन बन गया है।

### Cursor

Cursor MCP सर्वर को `mcp.json` से पढ़ता है—प्रोजेक्ट की प्रति `.cursor/mcp.json` पर या वैश्विक प्रति `~/.cursor/mcp.json` पर। उसके दस्तावेज़ों में रिमोट सर्वर की संरचना पर्यावरण चर के इंटरपोलेशन के साथ हेडर-आधारित प्रमाणीकरण का समर्थन करती है, इसलिए कुंजी को फ़ाइल में रखने की ज़रूरत नहीं है:

```json
{
  "mcpServers": {
    "namefi": {
      "url": "https://api.namefi.io/mcp",
      "headers": {
        "x-api-key": "${env:NAMEFI_API_KEY}"
      }
    }
  }
}
```

`${env:NAMEFI_API_KEY}` उस शेल में मौजूद इस चर के मान से हल होता है जिसने Cursor को शुरू किया था—एडिटर खोलने से पहले इसे एक्सपोर्ट करें।

### Windsurf (Cascade)

Windsurf का MCP एकीकरण—जिसका ब्रांड नाम Cascade है—`~/.codeium/windsurf/mcp_config.json` को पढ़ता है। वहाँ रिमोट सर्वर `serverUrl` फ़ील्ड इस्तेमाल करते हैं, `url` नहीं; `headers` और `${env:VAR}` का पैटर्न Cursor जैसा ही है:

```json
{
  "mcpServers": {
    "namefi": {
      "serverUrl": "https://api.namefi.io/mcp",
      "headers": {
        "x-api-key": "${env:NAMEFI_API_KEY}"
      }
    }
  }
}
```

एक बात ध्यान देने लायक है: इस गाइड की प्रकाशन तिथि पर `docs.windsurf.com/windsurf/cascade/mcp`, `docs.devin.ai/desktop/cascade/mcp` पर रीडायरेक्ट होता है—Windsurf के दस्तावेज़ अब Cognition के Devin उत्पाद-दस्तावेज़ डोमेन पर हैं और ऊपर दिया कॉन्फ़िगरेशन फ़ॉर्मेट उसी वर्तमान पृष्ठ में दर्ज है। यदि आपका बिल्ड पुराना है, तो फ़ील्ड के नाम उस दस्तावेज़ लिंक से मिलाएँ जिसकी ओर आपके संस्करण की इन-ऐप मदद संकेत करती है।

## पाँच-चरण वाला क्विकस्टार्ट: नए ऐप से लाइव DNS तक

ऊपर दिए कनेक्शन में से कोई एक चालू होने के बाद बाकी प्रवाह हर एडिटर में समान है।

1. **API कुंजी पाएँ**—इसे [namefi.io/api-key](https://namefi.io/api-key) पर उस वॉलेट से बनाएँ जिसे नए डोमेन का मालिक होना चाहिए।
2. **कनेक्ट करें**—ऊपर अपने एडिटर का कॉन्फ़िगरेशन इस्तेमाल करें, फिर एक सामान्य जाँच करें: पूछें, "जाँचें कि `<yourapp>.com` Namefi पर उपलब्ध है या नहीं और बताएँ कि आपने कौन-सा टूल कॉल किया।" यह केवल-पढ़ने वाली `checkAvailability` कॉल है, इसलिए किसी धनराशि की व्यवस्था करने से पहले भी काम करती है।
3. **रजिस्टर करें।** सामान्य भाषा में नाम और अवधि की पुष्टि करें—"इसे एक वर्ष के लिए रजिस्टर करें।" एजेंट `registerDomain` भेजता है और ऑर्डर को तब तक पोल करता है जब तक वह `SUCCEEDED` (या किसी अंतिम विफलता स्थिति) तक न पहुँच जाए; सामान्य रजिस्ट्रेशन कुछ पोल चक्रों में पूरा हो जाता है।
4. **इसे अपने डिप्लॉय की ओर पॉइंट करें।** अगले सेक्शन में इस चरण का विस्तार है—उसी बातचीत के ज़रिए आपके होस्टिंग प्लेटफ़ॉर्म द्वारा माँगे गए DNS रिकॉर्ड जोड़ें।
5. **पुष्टि करें कि यह रिज़ॉल्व होता है।** [DNS प्रोपेगेशन](/hi/glossary/dns-propagation/) तुरंत नहीं होता, इसलिए कुछ मिनट दें और फिर सार्वजनिक DNS लुकअप से या बस ब्राउज़र में डोमेन खोलकर पुष्टि करें।

## नए डोमेन को अभी किए गए डिप्लॉयमेंट की ओर पॉइंट करें

सामान्य "डोमेन कैसे रजिस्टर करें" गाइड इस हिस्से तक कभी नहीं पहुँचती, क्योंकि यह रजिस्ट्रेशन के बाद होस्टिंग प्लेटफ़ॉर्म की ओर होता है—लेकिन एडिटर के भीतर यह काम करने का असली उद्देश्य यही है: आपका एजेंट पहले से जानता है कि उसने किस प्लेटफ़ॉर्म पर डिप्लॉय किया और वह रजिस्ट्रेशन के साथ ही DNS जोड़ सकता है।

### Vercel

Vercel के अपने डोमेन दस्तावेज़ प्रोजेक्ट डैशबोर्ड में **Settings → Domains** से शुरू होने वाला प्रवाह बताते हैं: डोमेन जोड़ें और Vercel बताता है कि वह एपेक्स डोमेन है या सबडोमेन, उसके अनुसार कौन-सा रिकॉर्ड बनाना है। **एपेक्स डोमेन** (`yourapp.com`) के लिए Vercel अपने सर्विंग IP की ओर इशारा करता **A रिकॉर्ड** माँगता है; **सबडोमेन** (`www.yourapp.com`) के लिए वह **CNAME** माँगता है। किसी पुराने गाइड का उदाहरण कॉपी करने से पहले यह जानना उपयोगी है कि [Vercel के दस्तावेज़ स्पष्ट करते हैं कि यह CNAME लक्ष्य हर प्रोजेक्ट के लिए अलग होता है](https://vercel.com/docs/domains/working-with-domains/add-a-domain#:~:text=Each%20project%20has%20a%20unique%20CNAME%20record)—यह हर प्रोजेक्ट के साझा एक स्थिर होस्टनाम के बजाय आपको डैशबोर्ड में दिखाया जाता है।

वह मान मिलने के बाद DNS वाला हिस्सा एजेंट से एक और अनुरोध है:

> "`@` के लिए `76.76.21.21` की ओर पॉइंट करता A रिकॉर्ड और `www` के लिए Vercel द्वारा दिए गए CNAME लक्ष्य की ओर पॉइंट करता CNAME जोड़ें।"

यह `createDnsRecord` को दो बार कॉल करता है—हर रिकॉर्ड के लिए एक बार—और यह वही [DNS रिकॉर्ड](/hi/glossary/dns-record-types/) टूल है जो Namefi पर किसी भी DNS लेखन में इस्तेमाल होता है। यहाँ भी अंत में डॉट लगाने का वही नियम लागू होता है: CNAME लक्ष्य के `rdata` के अंत में डॉट चाहिए, जबकि `zoneName` (आपके डोमेन) के अंत में नहीं।

### Cloudflare Pages

यदि आपका डिप्लॉय लक्ष्य Cloudflare Pages है और आपके डोमेन का DNS पहले से Cloudflare पर प्रबंधित नहीं होता, तो [Cloudflare के अपने कस्टम-डोमेन दस्तावेज़](https://developers.cloudflare.com/pages/configuration/custom-domains/#:~:text=This%20record%20should%20point%20to%20your%20custom%20Pages%20subdomain) आपके प्रोजेक्ट के `.pages.dev` सबडोमेन की ओर पॉइंट करता एक **CNAME** रिकॉर्ड माँगते हैं—A रिकॉर्ड की ज़रूरत नहीं, क्योंकि Pages सब कुछ उस CNAME लक्ष्य से सर्व करता है। Cloudflare डैशबोर्ड वाला चरण (Workers & Pages → your project → Custom domains → Set up a domain) पहले करना होता है; उसके बाद ही CNAME लक्ष्य सही ढंग से रिज़ॉल्व होता है।

> "`app` के लिए `my-project.pages.dev.` की ओर पॉइंट करता CNAME जोड़ें।"

वही टूल कॉल, लक्ष्य पर अंत में डॉट लगाने का वही नियम, प्लेटफ़ॉर्म अलग।

## इसकी तुलना Cloudflare के इन-एडिटर रजिस्ट्रेशन से कैसे होती है

Cloudflare दूसरा रजिस्ट्रार है जो सक्रिय रूप से इन-एडिटर तरीके का प्रचार कर रहा है, इसलिए उसका सीधे उल्लेख करना उचित है। उसका Registrar API, [अप्रैल 2026 में बीटा होने की रिपोर्ट के अनुसार](https://webhosting.today/2026/04/22/ai-agents-can-now-register-domains-no-human-required/), Cursor और Claude Code सहित MCP-सक्षम एडिटर के साथ भी जुड़ता है। इससे एजेंट अपना मौजूदा कॉन्टेक्स्ट छोड़े बिना डोमेन खोजने, कीमत जानने और रजिस्ट्रेशन करने के काम समकालिक रूप से पूरा कर सकता है—यही मूल विचार यह गाइड Namefi के लिए समझाती है। उसी रिपोर्ट में कहा गया है कि बीटा के समय Cloudflare का API रजिस्ट्रेशन के बाद ट्रांसफ़र और रिन्यूअल जैसे प्रबंधन को अभी शामिल नहीं करता; इन्हें 2026 में बाद में लाने की योजना है।

Namefi का MCP सर्वर आज पूरा जीवनचक्र सँभालता है—रजिस्ट्रेशन, DNS और [ऑटो-रिन्यू](/hi/glossary/domain-renewal/)—साथ ही दो ऐसी चीज़ें भी देता है जो Cloudflare के रास्ते में नहीं हैं: डोमेन डिफ़ॉल्ट रूप से [टोकनाइज़्ड डोमेन](/hi/glossary/tokenized-domain/) NFT के रूप में रजिस्टर होता है (और उसे किसी भी वॉलेट पर रीडायरेक्ट किया जा सकता है), और यह बिना Namefi अकाउंट के वॉलेट से हस्ताक्षरित चेकआउट का समर्थन करता है, जिसका विवरण [क्रिप्टो वॉलेट से डोमेन के लिए भुगतान करें](/hi/blog/wallet-checkout/) में है। दोनों एक ही "एडिटर न छोड़ें" वर्कफ़्लो की ओर बढ़ रहे हैं; आपके लिए कौन-सा सही है, यह इस पर निर्भर है कि आपको सामान्य रजिस्ट्रेशन चाहिए या ऐसा रजिस्ट्रेशन जो ऑन-चेन एसेट भी हो।

## अक्सर पूछे जाने वाले प्रश्न

### क्या इसमें Codex या Gemini CLI भी शामिल हैं?

यहाँ नहीं—यह गाइड जानबूझकर Claude Code, Cursor और Windsurf तक सीमित है। [Namefi पर अपने AI एजेंट से डोमेन कैसे रजिस्टर करें](/hi/blog/ai-agent-register/) में Codex CLI, Gemini CLI और Claude Desktop के लिए ठीक यही सत्यापित कॉन्फ़िगरेशन मिलता है।

### क्या इसे आज़माने से पहले मुझे Namefi अकाउंट चाहिए?

नहीं। केवल-पढ़ने वाली उपलब्धता जाँच के लिए प्रमाणीकरण की ज़रूरत नहीं होती, इसलिए ऊपर दिया कोई भी एडिटर कनेक्ट करके API कुंजी बनाने या किसी धनराशि की व्यवस्था करने से पहले चरण 2 वाला परीक्षण प्रॉम्प्ट चला सकते हैं।

### यदि मेरा डिप्लॉयमेंट प्लेटफ़ॉर्म Vercel या Cloudflare Pages नहीं है तो क्या होगा?

पैटर्न हर जगह समान है: आपके प्लेटफ़ॉर्म का डैशबोर्ड बताता है कि उसे किस प्रकार का DNS रिकॉर्ड चाहिए—एपेक्स डोमेन के लिए लगभग हमेशा A रिकॉर्ड, सबडोमेन के लिए CNAME—और आप वह मान अपने एजेंट को `createDnsRecord` से लिखने के लिए देते हैं।

### क्या इस तरह रजिस्टर करने पर डोमेन अपने आप टोकनाइज़ हो जाता है?

हाँ, डिफ़ॉल्ट रूप से—डोमेन Base पर NFT के रूप में आपकी API कुंजी से जुड़े वॉलेट में रजिस्टर होता है, जब तक अनुरोध में कोई अलग `nftReceivingWallet` न दिया जाए। यदि यह आपके लिए नया है, तो [टोकनाइज़्ड डोमेन क्या हैं? डोमेन टोकनाइज़ेशन के लिए एक गाइड](/hi/blog/what-are-tokenized-domains/) देखें।

### क्या मैं API कुंजी को पूरी तरह छोड़ सकता हूँ?

हाँ, लेकिन एक शर्त के साथ: Namefi का वॉलेट से हस्ताक्षरित [x402](/hi/glossary/x402/) चेकआउट तरीका धन वाले वॉलेट को बिना किसी अकाउंट या API कुंजी के रजिस्ट्रेशन का भुगतान करने देता है। इसकी अलग व्याख्या चाहिए, जो [क्रिप्टो वॉलेट से डोमेन के लिए भुगतान करें](/hi/blog/wallet-checkout/) में दी गई है।

## ऐप के साथ नाम भी लाइव करें

डोमेन भी डिप्लॉय लक्ष्य और डेटाबेस जैसा इंफ़्रास्ट्रक्चर है—ऐप जारी करने का यही एक हिस्सा ऐसा रहे जिसके लिए अब भी अपने टूल छोड़कर वेब फ़ॉर्म भरना पड़े, इसका कोई वास्तविक कारण नहीं है। ऊपर दिए तीन कॉन्फ़िगरेशन में से एक कनेक्ट करें और पाँच चरणों वाला प्रवाह चलाएँ; एक भी ब्राउज़र टैब खोले बिना डोमेन लाइव होकर उसी डिप्लॉयमेंट की ओर पॉइंट करेगा जिसे आपके एजेंट ने अभी बनाया है।

**[Namefi API कुंजी बनाएँ](https://namefi.io/api-key)** और पहले से खुले अपने किसी भी एडिटर में उपलब्धता-जाँच वाला प्रॉम्प्ट आज़माएँ; या यदि आप हर चरण विस्तार से देखना चाहते हैं, तो [टिप्पणियों सहित ट्रांसक्रिप्ट वाला पूरा Claude Code चरण-दर-चरण गाइड](/hi/blog/claude-mcp-domains/) पढ़ें।
