Namefi MCP சேவையகத்துடன் இணையும் ஓர் [AI முகவருக்கு](/ta/glossary/ai-agent/), சேவையகம் ஆதரிக்கும் `/v-next` API பரப்பிலிருந்து உருவாக்கப்பட்ட அழைக்கக்கூடிய கருவிகள் கிடைக்கும். அவை தேடல், பதிவு, DNS, டொமைன் அமைப்பு, கணக்குச் செயல்பாடுகள் மற்றும் வெளிச்செல் சாத்திய வாடிக்கையாளர் தேடலை உள்ளடக்குகின்றன. x402 மற்றும் MPP கட்டணச் செயல்முறைகள் தனித்தனி HTTP முனைப்புள்ளிகளைப் பயன்படுத்துகின்றன; அவை MCP கருவிகள் அல்ல. இந்தப் பக்கம் ஆவணப்படுத்தப்பட்ட MCP பரப்பைப் பட்டியலிட்டு, அங்கீகாரத்தை விளக்கி, மூன்று பணிச்சுற்றுகளை வழங்குகிறது.

இதுவரை ஒரு முகவரை Namefi-யுடன் இணைக்கவில்லை எனில், ஒவ்வொரு கிளையண்டுக்குமான அமைப்புக்கு [Namefi-ல் உங்கள் AI முகவர் மூலம் ஒரு டொமைனைப் பதிவுசெய்வது எப்படி](/ta/blog/ai-agent-register/) என்பதிலிருந்து தொடங்குங்கள்; முழு உரையாடல் பதிவுக்கு [Claude மூலம் ஒரு டொமைன் வாங்குதல்: Namefi MCP படிப்படியான வழிகாட்டி](/ta/blog/claude-mcp-domains/) பார்க்கவும். இணைப்பு ஏற்கெனவே உள்ளது என்று இந்தப் பக்கம் கருதுகிறது.

## Namefi MCP சேவையகம் என்றால் என்ன

Namefi, `https://api.namefi.io/mcp`-ல் Streamable HTTP MCP சேவையகத்தை இயக்குகிறது. அரட்டையில் ஒட்டப்பட்ட ஆவணங்களை வைத்து ஒரு முகவர் தானாக REST அழைப்புகளை உருவாக்குவதற்குப் பதிலாக, அது இணைந்து, [api.namefi.io/v-next/openapi/doc.json](https://api.namefi.io/v-next/openapi/doc.json)-ல் உள்ள Namefi-ன் OpenAPI 3 விவரக்குறிப்பிலிருந்து உருவாக்கப்பட்ட, ஆதரிக்கப்படும் `/v-next` செயல்பாடுகளுக்கான வகையிடப்பட்ட கருவிகளைப் பெறுகிறது. இந்த உருவாக்க முறை கைமுறை முரண்பாட்டைக் குறைக்கிறது; ஆனால் நேரலைக் கருவிப் பட்டியல், OpenAPI ஆவணம், கண்டறிதல் விவரிப்பு மற்றும் வலைப்பதிவு நிலைப்படம் ஆகியவை வெவ்வேறு நேரங்களில் மாறக்கூடும். நீங்கள் உண்மையில் பயன்படுத்தும் இணைப்பின் `tools/list`-ஐச் சரிபாருங்கள்.

[namefi.io/.well-known/mcp/servers.json](https://namefi.io/.well-known/mcp/servers.json)-ல் உள்ள இயந்திரம் படிக்கக்கூடிய கண்டறிதல் விவரிப்பு, சேவையகத்தை `namefi-api` என்று பெயரிட்டு, `streamable-http` போக்குவரத்தை அறிவிக்கிறது. மேலும் PKCE, பாதுகாக்கப்பட்ட-வளக் கண்டறிதல், refresh tokens, device flow மற்றும் dynamic client registration உடனான `x-api-key` மற்றும் OAuth 2.1 authorization-code அங்கீகாரம் இரண்டையும் ஆவணப்படுத்துகிறது. [ICANN](/ta/glossary/icann/) அங்கீகாரம் பெற்ற [ரெஜிஸ்ட்ராரான](/ta/glossary/registrar/) Namefi, MCP-ஐ ஆதரிக்காத முகவர்களுக்கும் நிரல்களுக்கும் எளிய HTTPS முனைப்புள்ளிகளை [namefi.io/llms.txt](https://namefi.io/llms.txt)-லும் வெளியிடுகிறது. `/x402/...` மற்றும் `/mpp/...` என்பவை MCP கருவிகள் அல்ல, தனித்தனி HTTP கட்டண முனைப்புள்ளிகள் என்று அந்தக் கோப்பு வெளிப்படையாகக் குறிப்பிடுகிறது.

## முழுமையான திறன் பட்டியல்

ஆவணப்படுத்தப்பட்ட `/v-next` செயல்பாட்டுக் குழுக்களின் தேதி குறிப்பிடப்பட்ட நிலைப்படம் கீழே உள்ளது. **செயல்பாடு** நெடுவரிசை, MCP கருவியின் பெயர் உருவாக்கப்படும் OpenAPI `operationId`-ஐக் காட்டுகிறது. **அங்கீகாரம்** நெடுவரிசை REST நிலையிலான API-விசை வழியைக் காட்டுகிறது; அதற்குப் பதிலாக ஒரு MCP கிளையண்ட் தனது அமர்வை OAuth bearer token மூலம் அங்கீகரிக்கலாம். பொது REST என்று குறிக்கப்பட்ட வரிகள், பெயரில்லா MCP இணைப்பைக் குறிக்காது: அங்கீகரிக்கப்படாத `initialize` கோரிக்கை `401 Unauthorized` என்று பதிலளித்தது; இது **ஜூலை 14, 2026** அன்று சரிபார்க்கப்பட்டது.

### தேடலும் கண்டறிதலும்

| செயல்பாடு | முனைப்புள்ளி | அது செய்வது | அங்கீகாரம் |
| --- | --- | --- | --- |
| `checkAvailability` | `GET /v-next/search/availability` | ஒரு டொமைன் பெயரைப் பதிவுசெய்ய முடியுமா என்று சரிபார்க்கும் | பொது REST; தற்போது அங்கீகரிக்கப்பட்ட MCP அமர்வு தேவை |
| `checkBulkAvailability` | `GET /v-next/search/bulk-availability` | பல சாத்தியப் பெயர்களை ஒரே அழைப்பில் பரிசோதிக்கும் | பொது REST; தற்போது அங்கீகரிக்கப்பட்ட MCP அமர்வு தேவை |
| `getSuggestions` | `GET /v-next/search/suggestions` | ஒரு தேடலுடன் தொடர்புடைய கணிமுறைப் பெயர் பரிந்துரைகளைப் பெறும் | பொது REST; தற்போது அங்கீகரிக்கப்பட்ட MCP அமர்வு தேவை |

### பதிவும் ஆர்டர்களும்

| செயல்பாடு | முனைப்புள்ளி | அது செய்வது | அங்கீகாரம் |
| --- | --- | --- | --- |
| `registerDomain` | `POST /v-next/orders/register-domain` | ஒரு டொமைனை 0–10 ஆண்டுகளுக்குப் பதிவுசெய்யும். `domainSetupOptions` பொருளையும் (`autoPark`, `autoEns`, `autoRenew`, `dnssec`, `keepExistingNameservers`), விருப்பமான `nftReceivingWallet`-ஐயும் ஏற்கும் | API விசை |
| `registerWithRecords` | `POST /v-next/orders/register-domain/records` | ஒரே அழைப்பில் பதிவுசெய்து, தொடக்க DNS பதிவுகளையும் பயன்படுத்தும் | API விசை |
| `getOrder` | `GET /v-next/orders/{orderId}` | ஆர்டர் `SUCCEEDED`, `FAILED`, `CANCELLED` அல்லது `PARTIALLY_COMPLETED` என்ற இறுதி நிலையை அடையும் வரை நிலையைக் கேட்கும் | API விசை |

பதிவு ஒத்திசைவின்றி நடைபெறும் — `registerDomain` உடனடியாக ஆர்டர் `id`-ஐத் தரும்; அது முடிவடையும் வரை முகவர் `getOrder` மூலம் நிலையைக் கேட்டுக்கொண்டிருக்கும். [Claude வழிகாட்டி](/ta/blog/claude-mcp-domains/) மற்றும் [பல முகவர் அமைப்பு வழிகாட்டி](/ta/blog/ai-agent-register/) இரண்டிலும் இந்த முறை முழு உரையாடல் வடிவில் காட்டப்பட்டுள்ளது.

### DNS பதிவு நிர்வாகம்

ஒவ்வொரு பதிவாகவோ தொகுப்பாகவோ முழுமையான உருவாக்கம், வாசிப்பு, புதுப்பித்தல் மற்றும் நீக்குதல் கிடைக்கிறது. REST வாசிப்பு பொதுவானது என்று ஆவணப்படுத்தப்பட்டாலும், நேரலை MCP முனைப்புள்ளிக்கு தற்போது அமர்வே அங்கீகரிக்கப்பட்டிருக்க வேண்டும்:

| செயல்பாடு | முனைப்புள்ளி | அது செய்வது | அங்கீகாரம் |
| --- | --- | --- | --- |
| `getDnsRecords` | `GET /v-next/dns/records` | ஒரு zone-ல் உள்ள எல்லாப் பதிவுகளையும் பட்டியலிடும் | பொது REST; தற்போது அங்கீகரிக்கப்பட்ட MCP அமர்வு தேவை |
| `createDnsRecord` | `POST /v-next/dns/records` | ஒரு பதிவை உருவாக்கும் | API விசை |
| `updateDnsRecord` | `PUT /v-next/dns/record` | ID மூலம் ஒரு பதிவைப் புதுப்பிக்கும் | API விசை |
| `deleteDnsRecord` | `DELETE /v-next/dns/record` | ID மூலம் ஒரு பதிவை நீக்கும் | API விசை |
| `batchCreateDnsRecords` | `POST /v-next/dns/records/batch` | பல பதிவுகளை ஒரே அழைப்பில் உருவாக்கும் | API விசை |
| `batchUpdateDnsRecords` | `PUT /v-next/dns/records/batch` | பல பதிவுகளை ஒரே அழைப்பில் புதுப்பிக்கும் | API விசை |
| `batchDeleteDnsRecords` | `DELETE /v-next/dns/records/batch` | பல பதிவுகளை ஒரே அழைப்பில் நீக்கும் | API விசை |

ஆதரிக்கப்படும் [பதிவு வகைகள்](/ta/glossary/dns-record-types/): A, AAAA, CNAME, MX, TXT, NS, SOA, PTR, SRV, CAA, DS, TLSA, SSHFP, HTTPS, SVCB, NAPTR, SPF. முதல் முயற்சிகளில் பெரும்பாலும் தவறவிடப்படும் இரண்டு வடிவமைப்பு விதிகள்: `zoneName`-ன் முடிவில் புள்ளி இருக்கக் கூடாது; ஆனால் CNAME, MX மற்றும் NS பதிவுகளின் `rdata` மதிப்புகளின் முடிவில் புள்ளி இருக்க வேண்டும்.

### டொமைன் நிலையிலான மாற்றுகள்

தனிப்பட்ட DNS பதிவிலிருந்து வேறுபட்டு, இவை ஒரு முழு அம்சத்தை இயக்கவோ நிறுத்தவோ செய்கின்றன:

| செயல்பாடு | முனைப்புள்ளி | அது செய்வது | அங்கீகாரம் |
| --- | --- | --- | --- |

## அங்கீகாரம்: MCP இணைப்பு அங்கீகாரமும் REST கோரிக்கை அங்கீகாரமும் வெவ்வேறு அடுக்குகள்

ஒரு MCP கிளையண்ட் முதலில் சேவையக இணைப்பை அங்கீகரிக்க வேண்டும். தற்போதைய கண்டறிதல் விவரிப்பு, authorization code மற்றும் PKCE, பாதுகாக்கப்பட்ட-வள மற்றும் authorization-server metadata, refresh tokens, device flow, dynamic client registration ஆகியவற்றுடன் OAuth 2.1 bearer tokens-ஐ அறிவிக்கிறது. எனவே இணக்கமான கிளையண்டுகள் தங்கள் அமைப்பில் Namefi API விசையைச் சேமிக்காமல் MCP URL-உடன் இணைந்து OAuth செயல்முறையை நிறைவு செய்யலாம்.

**API விசை (`x-api-key`).** வெளிப்படையான header அடிப்படையிலான அல்லது தானியக்க அமைப்புகளுக்கு [namefi.io/api-key](https://namefi.io/api-key)-ல் விசையை உருவாக்கி, HTTP header ஆக அனுப்புங்கள். விசை உணர்திறன் கொண்டது: commit செய்யப்பட்ட MCP அமைப்பில் அதை வைக்காதீர்கள்; Namefi அதற்கு வழங்கும் அனுமதிகளுக்குள் மட்டுமே அதன் பயன்பாட்டை வரையறுக்கவும்.

சில படிக்க மட்டும் பயன்படும் கருவுகளுக்கு அங்கீகாரம் தேவையில்லை என்றும் விவரிப்பு கூறுகிறது. ஆனால் அங்கீகரிக்கப்படாத MCP `initialize` கோரிக்கை `401 Unauthorized` என்று பதிலளித்தது; இது **ஜூலை 14, 2026** அன்று சரிபார்க்கப்பட்டது. எந்தக் கருவி அழைப்புக்கும் முன்பே தொடக்கம் நடைபெறுவதால், செயலாக்கமும் விவரிப்பும் ஒன்றுபடும் வரை நேரலை MCP இணைப்புக்கு அங்கீகாரம் தேவை என்றே கருதுங்கள்.

நேரடி REST அழைப்புகளுக்கு, கூடுதல் கோரிக்கை-நிலை அங்கீகார முறைகளை Namefi ஆவணப்படுத்துகிறது. ஒவ்வொரு கருவி அழைப்புக்கும் MCP கிளையண்ட் தானாகப் பணப்பைக் கையொப்பத்தைப் பெறும் என்ற வாக்குறுதி இவை அல்ல:

**EIP-712 வகையிடப்பட்ட-தரவு கையொப்பம்.** சேமிக்கப்பட்ட விசையின்றி நிரலாக்கிப் பயன்படுத்த, ஒவ்வொரு கோரிக்கையையும் Ethereum [பணப்பை](/ta/glossary/wallet/) மூலம் கையொப்பமிடுங்கள்: `x-namefi-signer`, `x-namefi-signature`, `x-namefi-eip712-type` headers, payload-ஐ timestamp மற்றும் 300 வினாடிகளுக்குப் பிறகு காலாவதியாகும் ஒருமுறைப் பயன்பாட்டு nonce கொண்ட envelope-க்குள் பொதிகின்றன — API விசை இல்லாதபோது `toggleDomainParking`, `createDnsRecord`, `registerDomain` போன்ற செயல்பாடுகள் தேவைப்படுத்தும் முறை இதுதான். Namefi-ன் ஆவணங்களின்படி அவை மாறக்கூடும் என்பதால், domain மற்றும் type வரையறைகள் hardcode செய்யப்பட்ட மாறிலியிலிருந்து அல்லாமல் நேரலை முனைப்புள்ளிகளிலிருந்து (`GET /v-next/eip712/domain`, `/eip712/types`) பெறப்படுகின்றன. Smart-contract பணப்பைகள் நேரடியாகக் கையொப்பமிட முடியாது; எனவே அங்கீகரிக்கப்பட்ட externally-owned account, ஒப்பந்தத்தின் சார்பாகக் கையொப்பமிடுகிறது; கோரிக்கையை எந்த ஒப்பந்தம் அங்கீகரிக்கிறது என்பதை `x-namefi-erc1271-account` அல்லது `x-namefi-eip7702-account` குறிப்பிடும்.

**SIWE (Sign-In with Ethereum).** உரிமையுள்ள டொமைன்கள் அல்லது ஆர்டர்களைப் பட்டியலிடுவது போன்ற, ஒவ்வொரு அழைப்புக்கும் புதிய கையொப்பம் தேவையில்லாத பாதுகாக்கப்பட்ட REST வாசிப்புகளுக்கான அமர்வு token (`x-namefi-siwe-token`): nonce-ஐப் பெற்று, கையொப்பமிட வேண்டிய செய்தியைப் பெற்று, `personal_sign` மூலம் கையொப்பமிட்டு, அதைச் சரிபார்த்து, பின்னர் token-ஐ மீண்டும் பயன்படுத்துங்கள்.

x402 மற்றும் MPP செயல்முறைகள் தங்களது தனித்தனி HTTP முனைப்புள்ளிகளுக்கு கட்டண அங்கீகாரத்தைச் சேர்க்கின்றன; அங்கீகரிக்கப்படாத MCP இணைப்புக்கான மாற்று அங்கீகார முறைகள் அவை அல்ல. அந்த HTTP செயல்முறைகளைத் தொடக்கம் முதல் முடிவு வரை [கிரிப்டோ பணப்பை மூலம் டொமைன்களுக்குப் பணம் செலுத்துங்கள்: கணக்கு தேவையில்லை](/ta/blog/wallet-checkout/) விளக்குகிறது.

## டோக்கனைசேஷன் பட்டியலின் ஊடாகவே செயல்படுகிறது; அதன் அருகே தனியாக அல்ல

`registerDomain`, நியமிக்கப்பட்ட பெறுநர் பணப்பைக்கு Namefi ஆதரிக்கும் சங்கிலியில் டொமைனை ஓர் [NFT](/ta/glossary/nft/) ஆக உருவாக்க முடியும் — அது பல பணப்பைகளும் சந்தைகளும் காட்டக்கூடிய [தரநிலை இடைமுகமான](https://eips.ethereum.org/EIPS/eip-721) ஓர் [ERC-721](/ta/glossary/erc-721/) token. அந்தப் பணப்பையை `nftReceivingWallet` தேர்ந்தெடுக்கிறது; அது விரும்பிய எந்தச் சங்கிலியையும் தேர்ந்தெடுக்காது. Namefi அமைப்புக்குள் ஆன்செயின் கட்டுப்பாட்டை அந்த token பிரதிநிதித்துவப்படுத்துகிறது; ரெஜிஸ்ட்ரார், பதிவகம், ICANN கொள்கை, Namefi-ன் ஒப்பந்தங்கள் மற்றும் நீதிமன்ற உத்தரவுகள் ஆகியவை தனித்தனி அடுக்குகளாகவே இருக்கின்றன. சந்தையில் காட்டுவதும் வர்த்தக ஆதரவும் தளம் மற்றும் collection அமைப்பைப் பொறுத்தவை.

## மூன்று முகவர்கள், ஒரே கருவித் தொகுப்பைப் பயன்படுத்த மூன்று வழிகள்

**ஒரு உருவாக்குநர் ஒரே உரையாடலில் டொமைனைப் பதிவுசெய்து DNS-ஐ வெளியிடுகிறார்.** பெயர் காலியாக உள்ளதை `checkAvailability` உறுதிப்படுத்துகிறது; `registerDomain`, `domainSetupOptions`-ல் `autoRenew` மற்றும் `dnssec` ஆகியவற்றை அமைத்து அதைச் சமர்ப்பிக்கிறது; ஆர்டர் `SUCCEEDED` நிலையை அடைந்ததும், வெளியீட்டுத் தளத்தின் சரிபார்ப்புப் படி காத்திருக்கும் CNAME மற்றும் TXT பதிவுகளை `batchCreateDnsRecords` எழுதுகிறது. இந்த வரிசையை எடிட்டருக்குள் [கோடிங் முகவர்களுக்கான Namefi MCP விரைவுத் தொடக்கம்](/ta/blog/mcp-quickstart/) விளக்குகிறது.

**ஒரு டொமைன் வர்த்தகர் தொகுப்பை நிர்வகிக்கிறார்.** தற்போதைய சொத்துகளை `getUserDomains` பெறுகிறது; புதிய சாத்தியப் பெயர்களை `checkBulkAvailability` ஒரே அழைப்பில் பரிசோதிக்கிறது; வாங்கத் தகுந்தவற்றை `registerDomain` பெறுகிறது. மறுவிற்பனை செய்யப்படும் பெயர்களுக்கு `toggleDomainParking` ஓர் இறங்குப் பக்கத்தை அமைக்கிறது; அது நேரலையில் இருப்பதை `isDomainParked` உறுதிசெய்கிறது. தொகுப்பு முழுவதிலும், எந்தப் பெயர்கள் தொடர்ச்சியான புதுப்பித்தல் அங்கீகாரத்திற்குத் தகுந்தவை, எந்த ஊகப் பெயர்கள் காலாவதியாக அனுமதிக்கப்படலாம் என்பதை `getAutoRenew` மற்றும் `toggleAutoRenew` முடிவுசெய்கின்றன.

**ஒரு நிறுவனம் ஏற்கெனவே சொந்தமாக வைத்துள்ள பெயர்களுக்கு வெளிச்செல் சாத்திய வாடிக்கையாளர் தேடலை இயக்குகிறது.** பயன்படுத்தப்படாத டொமைனை `getUserDomains` கண்டறிகிறது; ஆராய்ச்சியை `startOutboundRun` தொடங்குகிறது; `getOutboundRun`, அதன் நிலை `SUCCEEDED` ஆகும் வரை கேட்டுக்கொண்டிருக்கும். அந்தப் பெயரை விரும்பக்கூடும் எனத் தொழில் சுயவிவரம் காட்டும் தரவரிசைப்படுத்தப்பட்ட நிறுவனங்களை `listOutboundLeads` தருகிறது; ஒவ்வொரு வாய்ப்புக்கும் `prepareOutboundOutreach` மின்னஞ்சல் வரைவினை உருவாக்குகிறது — ஒருமுறை உருவாக்கப்பட்ட பிறகு, மீண்டும் அழைக்கும்போது இலவசமாகத் திருப்பித் தரப்படும்.

## முகவர் இவற்றில் எதையும் மேற்பார்வையின்றி இயக்குவதற்கு முன்

`registerDomain`, `registerWithRecords`, `startOutboundRun`, `prepareOutboundOutreach` ஆகிய நான்கு செயல்பாடுகளும் **முக்கிய விளைவுகளை ஏற்படுத்துபவை** என்று Namefi-ன் சொந்த வெளிச்செல் ஆவணம் குறிக்கிறது; ஏனெனில் ஒவ்வொன்றும் இருப்பிலிருந்து செலவிடுகிறது அல்லது வெளியில் புலப்படும் நடவடிக்கையை எடுக்கிறது. `checkAvailability` போன்ற படிக்க மட்டும் பயன்படும் அழைப்பு பணம் செலவிடாது; ஆனால் தற்போதைய MCP அமர்வுக்கு இன்னும் அங்கீகாரம் தேவை, மேலும் அதன் வெளியீடு பின்னர் வரும் செயல்களைப் பாதிக்கலாம். ஆர்டரை எழுதுவது, நேரலை டொமைனின் DNS-ஐ மாற்றுவது அல்லது வெளிப்புற நபருக்கான அணுகலை உருவாக்குவது எதுவாக இருந்தாலும், வெளிப்படையான உறுதிப்படுத்தல் படி தேவை. எந்த ரெஜிஸ்ட்ராரின் முகவர் சார்ந்த பரப்பையும் இவ்வாறு மதிப்பிடுவதற்கான விரிவான சரிபார்ப்புப் பட்டியலை [முகவரை மையமாகக் கொண்ட டொமைன் ரெஜிஸ்ட்ரார் என்றால் என்ன?](/ta/blog/agent-native/) வழங்குகிறது.

## இந்தப் பட்டியலைப் புதுப்பித்த நிலையில் வைத்திருத்தல்

மேலுள்ள வெளியீட்டுத் தேதியின்படி Namefi-ன் நேரலை OpenAPI விவரக்குறிப்பை இந்த அட்டவணை பிரதிபலிக்கிறது; இது நிலையான திட்ட வரைபடம் அல்ல — புதிய செயல்பாடுகள் எந்த வலைப்பதிவின் அட்டவணையையும் விட முன்பாக [namefi.io/llms.txt](https://namefi.io/llms.txt) மற்றும் [namefi.io/llms-full.txt](https://namefi.io/llms-full.txt)-ல் சேர்கின்றன.

## அடிக்கடி கேட்கப்படும் கேள்விகள்

### ஒரு பெயர் கிடைக்கிறதா என்று மட்டும் சரிபார்க்க எனக்கு API விசை தேவையா?
கட்டாயமில்லை. நிலையான API விசைக்குப் பதிலாக இணக்கமான MCP கிளையண்ட் OAuth மூலம் அங்கீகரிக்கலாம். அடிப்படையான கிடைக்கும்தன்மை REST செயல்பாடுகள் பொதுவானவை என்று ஆவணப்படுத்தப்பட்டுள்ளன; ஆனால் ஜூலை 14, 2026 அன்று சரிபார்க்கப்பட்டபோது, தொடங்குவதற்கே நேரலை MCP முனைப்புள்ளி அங்கீகாரம் தேவைப்படுத்தியது. கிடைக்கும்தன்மைச் சரிபார்ப்புகளுக்கு நிதி தேவையில்லை.

### நான் Namefi API விசையை எப்போதும் வைத்திருக்காமலேயே ஒரு முகவர் இந்தப் பட்டியல் முழுவதையும் பயன்படுத்த முடியுமா?
Namefi API விசையைச் சேமிக்காமல், MCP இணைப்புக்கு OAuth-ஐப் பயன்படுத்த முடியும். நேரடி REST ஒருங்கிணைப்புக்கு ஆதரிக்கப்படும் கோரிக்கை வகைகளை EIP-712 மற்றும் SIWE உள்ளடக்குகின்றன. x402 மற்றும் MPP பதிவுச் செயல்முறைகள் தனித்தனி HTTP முனைப்புள்ளிகள்; அவை MCP கருவிகளோ MCP இணைப்பு அங்கீகாரத்திற்கான மாற்றுகளோ அல்ல. எனவே உங்களுக்குத் தேவையான குறிப்பிட்ட செயல்பாட்டிற்கான தற்போதைய ஆவணத்தைச் சரிபாருங்கள்.

### இந்த வழிகளில் எதன் மூலமாகப் பதிவுசெய்தாலும் டொமைன் தானாகவே டோக்கனைஸ் செய்யப்படுமா?
ஆவணப்படுத்தப்பட்ட `/v-next` `registerDomain` செயல்முறையில், நியமிக்கப்பட்ட பெறுநர் பணப்பைக்கு Namefi தனது ஆதரிக்கப்படும் சங்கிலியில் டொமைன் token-ஐ உருவாக்குகிறது. ஒவ்வொரு தனிப்பட்ட பதிவு மற்றும் கட்டண வழிக்கும் ஒரே இயல்புநிலைகள் இருப்பதாக இந்தப் பட்டியல் நிறுவவில்லை; எனவே API-விசைப் பணப்பையையோ ஒரு குறிப்பிட்ட சங்கிலியையோ ஊகிக்காமல், நீங்கள் அழைக்கும் வழிக்கான தற்போதைய ஆவணத்தைச் சரிபாருங்கள்.

### தன்னாட்சி முகவர் இயக்குவதற்கு முன் ஒரு மனிதர் எந்தச் செயல்பாடுகளை உறுதிப்படுத்த வேண்டும்?
குறைந்தபட்சம், Namefi-ன் ஆவணங்கள் முக்கிய விளைவுகளை ஏற்படுத்துபவை எனக் குறிக்கும் நான்கையும் — `registerDomain`, `registerWithRecords`, `startOutboundRun`, `prepareOutboundOutreach` — மேலும் ஏற்கெனவே நேரலைப் போக்குவரத்தை வழங்கும் டொமைனில் செய்யப்படும் எந்த DNS எழுதுதலையும் மனிதர் உறுதிப்படுத்த வேண்டும்.

## தற்போதைய பட்டியலுடன் உங்கள் முகவரை இணைக்கவும்

ஆதரிக்கப்படும் `/v-next` பரப்பிலிருந்து உருவாக்கப்படும் MCP கருவிகள் `https://api.namefi.io/mcp` மூலம் கிடைக்கின்றன; x402 மற்றும் MPP தனித்தனி HTTP முனைப்புள்ளிகளாகவே உள்ளன. இந்தப் பட்டியல் மாறக்கூடும் என்பதால், இக்கட்டுரையை நிலையான schema ஆகக் கருதாமல், இணைக்கப்பட்ட சேவையகத்தின் `tools/list`-ஐச் சரிபாருங்கள். இதுவரை இணைக்கவில்லை எனில், ஆறு கிளையண்டுகளுக்கான அமைப்பை [Namefi-ல் உங்கள் AI முகவர் மூலம் ஒரு டொமைனைப் பதிவுசெய்வது எப்படி](/ta/blog/ai-agent-register/) விளக்குகிறது; அதன் கீழுள்ள கண்டறிதல் அடுக்கை [டொமைன்களுக்கான llms.txt](/ta/blog/llms-txt/) விளக்குகிறது.

இணக்கமான கிளையண்டில் OAuth மூலம் இணைக்கவும்; அல்லது வெளிப்படையான header அடிப்படையிலான அமைப்புக்காக **[Namefi API விசையை உருவாக்கவும்](https://namefi.io/api-key)**; பின்னர் நேரலைக் கருவிப் பட்டியலைச் சரிபார்க்கவும்.
