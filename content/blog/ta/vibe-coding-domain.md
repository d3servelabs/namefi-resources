நீங்கள் ஓர் அறிவுறுத்தலைத் தட்டச்சு செய்தீர்கள், கோப்புகளின் வரிசை நிரம்புவதைப் பார்த்தீர்கள், முப்பது வினாடிகளுக்குப் பிறகு உரையாடலில் ஒரு நேரடி URL தோன்றியது. வைப் கோடிங்கின் கவர்ச்சி இதுதான்: “என்னிடம் ஓர் யோசனை இருக்கிறது” என்பதற்கும் “இணையத்தில் செயல்படும் ஒன்று இருக்கிறது” என்பதற்கும் இடையிலான காலம், ஒரு காபி இடைவேளையின் அளவுக்குச் சுருங்கிவிட்டது. ஆனால் நீங்கள் பார்க்கும் URL, `my-app-a3f9.vercel.app` அல்லது `my-app.lovable.app` போன்ற ஒன்றில் முடிகிறது — அது ஒரு தளத் துணைடொமைன்; வணிக அட்டையில் அச்சிடக்கூடிய பெயர் அல்ல. அங்கிருந்து உண்மையில் நீங்கள் கட்டுப்படுத்தும் டொமைனுக்குச் செல்லும்போதுதான் வழக்கமாகப் பணியோட்டம் துண்டிக்கிறது; அப்படி நடக்க வேண்டிய அவசியமில்லை.

## “வைப் கோடிங்” உண்மையில் என்றால் என்ன

இந்தச் சொல்லை நீங்கள் ஏற்கெனவே அறிந்திருக்கவில்லை என்றால்: [Wikipedia, வைப் கோடிங்கை வரையறுக்கும் விதம்](https://en.wikipedia.org/wiki/Vibe_coding) இதுதான் — “செயற்கை நுண்ணறிவின் (AI) உதவியுடன் மேற்கொள்ளப்படும் ஒரு மென்பொருள் உருவாக்க நடைமுறை; இதில் மென்பொருள் உருவாக்குநர் ஒரு திட்டம் அல்லது பணியைப் பெரிய மொழி மாதிரிக்கு (LLM) அறிவுறுத்தல் மூலம் விவரிக்கிறார், அது தானாகவே மூல நிரலை உருவாக்குகிறது.” இதன் வரையறுக்கும் பண்பு AI நிரலை எழுதுவது மட்டுமல்ல — பழைய பல கருவிகள் தானியங்கி நிரப்பல் மூலம் அதைச் செய்தன — மாதிரி உருவாக்கிய ஒவ்வொரு வரியையும் படிப்பதற்குப் பதிலாக, அது அளிக்கும் முடிவை நீங்கள் பெரும்பாலும் ஏற்றுக்கொண்டு, அடுத்த மாற்றத்தை இயல்பான மொழியில் விவரித்து மீண்டும் மீண்டும் மேம்படுத்துவதுதான். Tesla-வின் முன்னாள் AI தலைவரும் OpenAI இணை நிறுவனருமான Andrej Karpathy, பிப்ரவரி 2025-ல் இந்தச் சொல்லை உருவாக்கினார்; ஒரே மாதத்துக்குள் Merriam-Webster அதை வேகமாகப் பரவும் வழக்குச் சொல்லாகக் குறிப்பிட்டு, பின்னர் Collins English Dictionary அதை அந்த ஆண்டின் சொற்களில் ஒன்றாகத் தேர்ந்தெடுக்கும் அளவுக்கு அது விரைவாகப் பரவியது.

இது அந்த நடைமுறையைக் குறைகூறுவது அல்ல. நீங்கள் விரும்புவதை விவரித்து, இயங்கும் செயலியைத் திரும்பப் பெறுவது உண்மையிலேயே ஒரு புதிய உருவாக்க முறை. இதைச் சுற்றி உருவான Cursor, Lovable, Replit, bolt.new, v0, Claude Code போன்ற கருவிகள், இயங்கும் மாதிரி செயலியை உருவாக்குவது இனி கடினமான பகுதி அல்ல என்ற அளவுக்கு மேம்பட்டுள்ளன. கடினமான பகுதி — அல்லது குறைந்தபட்சம் இன்னும் 2015-ல் இருப்பதுபோல் தோன்றும் பகுதி — “இது இயங்குகிறது” என்பதற்குப் பிறகு வரும் அனைத்தும்: அதற்குப் பெயரிடுவதும், உண்மையான முகவரியை வழங்குவதும்.

## கடைசி கட்டம்: தளத் துணைடொமைனிலிருந்து உங்கள் சொந்த டொமைனுக்கு

ஹோஸ்டிங் தளங்கள் பொதுவாக முதலில் தங்கள் சொந்த டொமைனின் [துணைடொமைனில்](/ta/glossary/subdomain/) வெளியிடவும், பின்னர் தனிப்பயன் டொமைனை அமைக்கவும் அனுமதிக்கின்றன. Cursor அல்லது Claude Code போன்ற திருத்திகளும் கோடிங் முகவர்களும் ஒவ்வொரு செயலியையும் தாங்களே ஹோஸ்ட் செய்வதில்லை; எனவே சரியான வெளியீட்டு URL, அந்தத் திட்டம் பயன்படுத்தும் ஹோஸ்டிங் சேவையிலிருந்து வருகிறது. தளத் துணைடொமைன் சோதனைக்குப் பயனுள்ளதாக இருக்கும்; ஆனால் நீங்கள் கட்டுப்படுத்தும் டொமைனைவிட நினைவில் வைத்துக்கொள்வது கடினமாக இருக்கலாம்.

உண்மையான டொமைனைப் பதிவுசெய்வது தனிப்பட்ட முறையில் சிறிய பணிதான் — பெயரைத் தேடுதல், வாங்குதல், சில DNS பதிவுகளை அமைத்தல் — ஆனால் முழு வைப் கோடிங் சுழற்சியிலும் வழக்கமாக முற்றிலும் வேறு இடத்தில் நடக்கும் ஒரே படி அதுதான்.

## திருத்தியை விட்டு வெளியேறுவது பணியோட்டத்தை ஏன் துண்டிக்கிறது

உண்மையான சிரமம் டொமைன் பதிவு கடினமானது என்பதல்ல. அது *வேறொரு இடத்தில்* இருக்கிறது என்பதுதான். வழக்கமான முறையில் ஒரு டொமைனைப் பதிவுசெய்ய, உங்கள் கோடிங் முகவருடன் நடத்தும் உரையாடலை நிறுத்தி, உலாவித் தாவல் ஒன்றைத் திறந்து, ஒரு [ரெஜிஸ்ட்ராரின்](/ta/glossary/registrar/) முகப்புப் பக்கத்திற்குச் சென்று, பெயரைத் தேடி, உங்களுக்குத் தேவையில்லாத தனியுரிமைப் பாதுகாப்பு, மின்னஞ்சல் ஹோஸ்டிங், வலைத்தள உருவாக்கி ஆகிய மூன்று கூடுதல் விற்பனைச் சலுகைகளைப் பார்த்து, எந்தத் தேர்வுப்பெட்டியின் குறியை நீக்க வேண்டும் என்று கண்டறிந்து, பணம் செலுத்த வேண்டும். அதன் பிறகு — பொதுவான டொமைன் வழிகாட்டிகள் தவிர்க்கும் பகுதி இதுதான் — உங்கள் குறிப்பிட்ட ஹோஸ்டிங் தளம் எந்த [DNS](/ta/glossary/dns/) பதிவைக் கோருகிறது என்பதைக் கண்டறிந்து, அதன் மதிப்பை வேறொரு கட்டுப்பாட்டுப் பலகத்தில் தேடி, மூன்றாவது தாவலில் ஒட்ட வேண்டும்.

அது ஒரே பணி அல்ல; ஐந்து பணிகள். அவை மூன்று வெவ்வேறு தயாரிப்புகளில் பரவியுள்ளன; அவற்றில் எதற்கும் நீங்கள் இப்போது உருவாக்கியது என்ன, எந்தத் தளத்தில் வெளியிட்டீர்கள் என்பது தெரியாது. ஒவ்வொரு சூழல் மாற்றத்திற்கும் உண்மையான விலை உண்டு: நீங்கள் செய்துகொண்டிருந்த பணியின் தொடர்ச்சியை இழக்கிறீர்கள்; மேலும் அந்த மற்ற தாவல்களில் ஏதோ ஒன்றால் கவனம் சிதறி, ஒரு மணி நேரத்துக்குப் பிறகு திரும்பிவரும் வாய்ப்பும் உண்டு. ஐந்து நிமிடப் பணிக்கு இது மிக அதிகமான மேலதிகச் சுமை.

## உரையாடலை விட்டு வெளியேறாமல் பதிவுசெய்யுங்கள்

தீர்வு என்னவென்றால், வெளியீட்டை அணுகுவதுபோலவே டொமைனையும் அணுக வேண்டும்: வேறொரு வேலையாக அல்லாமல், அதே உரையாடலுக்குள் இன்னொரு கருவி அழைப்பாகக் கருதுங்கள். உங்கள் செயலிக்கான கட்டமைப்பை உருவாக்கி வெளியீட்டை அனுப்பிய முகவருக்கு ஏற்கெனவே சூழல் தெரியும் — செயலியின் பெயர், அது இயங்கும் தளம் — எனவே பெயரைச் சரிபார்த்து, பதிவுசெய்து, DNS-ஐ இணைப்பதற்கும் அதுவே சரியான கருவி.

முக்கிய அம்சங்களாகச் சுருக்கினால், பணியோட்டத்தில் மூன்று படிகள் உள்ளன:

1. **பெயரைச் சரிபார்க்க முகவரைக் கேளுங்கள்.** “`myapp.com` கிடைக்கிறதா?” என்பது படிக்க மட்டும் செய்யும் அழைப்பு; இது பணத்தைச் செலவிடாது. தற்போதைய நேரடி MCP endpoint-க்கு, தொடக்கநிலைப்படுத்தலின்போது OAuth அல்லது API key மூலம் அமர்வு அங்கீகரிக்கப்பட வேண்டியது இன்னும் அவசியம்.
2. **உறுதிசெய்து பதிவுசெய்யுங்கள்.** “ஒரு ஆண்டுக்கு இதைப் பதிவுசெய்” என்பது கோரிக்கையைச் சமர்ப்பிக்கும்; அது முடியும் வரை முகவர் கண்காணிக்கும்.
3. **உங்கள் வெளியீட்டை நோக்கிச் சுட்டுங்கள்.** உங்கள் ஹோஸ்டிங் தளம் கோரும் துல்லியமான DNS பதிவுகளை முகவருக்கு வழங்குங்கள்; Namefi நிர்வகிக்கும் மண்டலத்தில் ஆதரிக்கப்படும் அந்தப் பதிவுகளை அது எழுதும். பதிவு வகைகளும் மதிப்புகளும் ஹோஸ்டிங் வழங்குநரைப் பொறுத்து மாறும்; ஒவ்வொரு apex-மும் A record-ஐப் பயன்படுத்தும் என்றோ, ஒவ்வொரு துணைடொமைனும் CNAME-ஐப் பயன்படுத்தும் என்றோ கருதாதீர்கள். இங்கு விவரிக்கப்பட்டுள்ள தற்போதைய Namefi MCP/API பணியோட்டம், parent-zone [nameserver](/ta/glossary/nameserver/) delegation மாற்றங்களை வழங்குவதில்லை.

இதுதான் அதன் வடிவம். ஒவ்வொரு திருத்தியும் எந்த config கோப்பைப் படிக்கிறது, Vercel மற்றும் Cloudflare Pages கோரும் சரியான DNS மதிப்புகள் என்ன ஆகிய துல்லியமான செயல்முறைகள் [Namefi MCP Quickstart: Claude Code, Cursor & Windsurf](/ta/blog/mcp-quickstart/) கட்டுரையில் ஏற்கெனவே படிப்படியாக விளக்கப்பட்டுள்ளன; எனவே இங்கு அவற்றை மீண்டும் கூறமாட்டோம். அந்த மூன்று திருத்திகளைத் தவிர — OpenAI Codex, Gemini CLI, Claude Desktop அல்லது [MCP](https://modelcontextprotocol.io) பேசும் வேறெதிலும் — நீங்கள் கோடிங் செய்தால், [Namefi-ல் உங்கள் AI முகவர் மூலம் ஒரு டொமைனைப் பதிவுசெய்வது எப்படி](/ta/blog/ai-agent-register/) என்ற மையக் கட்டுரையில் ஒவ்வொன்றுக்கும் சரிபார்க்கப்பட்ட அமைப்பும், MCP-ஐ ஆதரிக்காதவற்றுக்கான நேரடி REST வழியும் உள்ளது.

## பெயர் யோசனைகளுக்கும் முகவரைப் பயன்படுத்துங்கள்

பெயரிடும் படியும் தனியாகக் குறிப்பிடப்பட வேண்டியது; ஏனெனில் கொள்முதலை நிறைவுசெய்வதுபோலவே இதுவும் பணியோட்டத்தைத் துண்டிக்கிறது. வழக்கமான முறை: ஒரு பெயரை யோசிக்கவும், ரெஜிஸ்ட்ரார் தாவலுக்குச் சென்று அது ஏற்கெனவே எடுக்கப்பட்டிருப்பதைக் கண்டறியவும், வேறொரு பெயரை யோசிக்கவும், மீண்டும் தாவல் மாறவும் — ஏதாவது ஒன்று கிடைக்கும் வரை அல்லது கைவிட்டு இறுதியில் ஒரு எண்ணைச் சேர்க்கும் வரை இதைத் தொடரவும்.

Namefi-ன் API பல பெயர்களின் கிடைப்புத்தன்மையை ஒரே நேரத்தில் சரிபார்க்கும் வசதியை வழங்குகிறது — ஒவ்வொரு முகவரும் படிக்கும் அதே [namefi.io/llms.txt](https://namefi.io/llms.txt#:~:text=or%20screen%20many%20names%20at%20once) ஆவணம், இதை “பல பெயர்களை ஒரே நேரத்தில் பரிசோதிக்கும்” வழியாக விவரிக்கிறது. எனவே பெயர்களை ஒவ்வொன்றாகச் சோதிப்பதற்குப் பதிலாக, முகவருக்கு முழுப் பட்டியலை வழங்கி, உண்மையில் எவை கிடைக்கின்றன என்பதை ஒரே கோரிக்கை-பதில் சுற்றில் பெறலாம். நடைமுறையில் இது பெயரிடுதலை ஒரே அறிவுறுத்தலாக மாற்றுகிறது: “இந்தச் செயலி Streaky என்ற பழக்கக் கண்காணிப்பான் — `streaky.com`, `streaky.app`, `getstreaky.com`, `streaky.io` ஆகியவற்றைச் சரிபார்த்து, எவை கிடைக்கின்றன என்று சொல்.” முகவர் அந்தத் தொகுப்பை இயக்கி, முடிவைத் தெரிவிக்கிறது; ஏற்கெனவே பதிவுசெய்யப்பட்ட பெயரைக் காதலித்து விடுவதற்குப் பதிலாக, உண்மையில் நீங்கள் பெறக்கூடிய பெயர்களிலிருந்து தேர்ந்தெடுக்கிறீர்கள்.

## செயல்படும் எடுத்துக்காட்டு: அறிவுறுத்தலிலிருந்து நேரடி URL வரை

ஒரு சிறிய கருவியை வைப் கோடிங் செய்து ஓர் பிற்பகலைக் கழித்தீர்கள் என்று வைத்துக்கொள்வோம் — பகிரப்பட்ட மளிகைப் பட்டியல் செயலி; ஏற்கெனவே உள்ளவை உங்களை எரிச்சலூட்டியதால் உருவாக்கியது. அது தளத் துணைடொமைனில் நேரலையில் இருக்கிறது, இயங்குகிறது, சில நண்பர்கள் இணைப்பைக் கேட்கிறார்கள். அதே உரையாடல் சாளரத்தில் அமர்வின் மீதிப்பகுதி முழுவதும் இப்படித்தான் நடக்கும்:

`cartly.app` கிடைக்கிறதா என்று கேட்கிறீர்கள். அது கிடைக்கிறது. “இதை ஓர் ஆண்டுக்குப் பதிவுசெய்து, நாம் இப்போது வெளியிட்டதுடன் இணை” என்று சொல்கிறீர்கள். முகவர் பதிவைச் சமர்ப்பித்து, அது முடியும் வரை சோதிக்கிறது. பின்னர் நீங்கள் இப்போது வாங்கிய டொமைனுக்கு எந்த DNS பதிவு வேண்டும் என்று உங்கள் ஹோஸ்டிங் தளத்திடம் கேட்கிறது — அதன் கட்டுப்பாட்டுப் பலகத்தை ஒருமுறை பார்ப்பதன் மூலம் — இந்த நிகழ்வில் நீங்கள் `www` துணைடொமைனுக்குப் பதிலாக apex டொமைனைப் பயன்படுத்துவதால் அது ஓர் A record. அந்த மதிப்பை நீங்கள் மீண்டும் ஒட்டுகிறீர்கள்; முகவர் பதிவை எழுதுகிறது; சில நிமிடங்களுக்குப் பிறகு — DNS பரவுவதற்குச் சிறிது நேரம் தேவை — `cartly.app`, உங்கள் நண்பர்கள் வேறொரு தாவலில் ஏற்கெனவே திறந்து வைத்துள்ள அதே செயலியைத் திறக்கிறது. திருத்தியை விட்டு வெளியே செலவிட்ட மொத்த நேரம்: பூஜ்யம். செயலியை உருவாக்குவதில் ஏற்கெனவே பயன்படுத்தாத புதிய தாவல்களின் மொத்த எண்ணிக்கை: பூஜ்யம்.

## அடிக்கடி கேட்கப்படும் கேள்விகள்

### இதைச் செய்ய எனக்கு DNS தெரிய வேண்டுமா?
ஒரு தரவுத்தளக் குறியீட்டைப் பயன்படுத்த, அது எவ்வாறு இயங்குகிறது என்பதை நீங்கள் அறிந்திருக்க வேண்டிய அளவுக்குத்தான். உங்கள் முகவர், ஹோஸ்டிங் தளத்திடம் எந்தப் பதிவு தேவை என்று கேட்டு அதை எழுதுகிறது; பெரும்பாலும் மதிப்புகளை நீங்கள் உறுதிசெய்கிறீர்கள், கையால் உருவாக்குவதில்லை.

### இது எந்த வைப் கோடிங் தளத்துடனும் இயங்குமா, அல்லது குறிப்பிட்ட சிலவற்றுடன் மட்டுமா?
பதிவும் DNS பகுதியும் தளம் சாராதவை — எந்தக் கருவி உங்கள் செயலியை உருவாக்கியது என்பதைப் பொருட்படுத்தாமல், டொமைனும் DNS பதிவும் ஒரேபோல இயங்குகின்றன. உங்கள் ஹோஸ்டிங் தளம் எந்தப் பதிவு வகையைக் கோருகிறது என்பதுதான் மாறும்; Vercel மற்றும் Cloudflare Pages-க்கு உரியவற்றை [Namefi MCP Quickstart](/ta/blog/mcp-quickstart/) குறிப்பாக விளக்குகிறது.

### இவ்வாறு நான் பதிவுசெய்யும் டொமைன் டோக்கன் ஆக்கப்படுமா?
ஆவணப்படுத்தப்பட்ட `/v-next` `registerDomain` பணியோட்டத்தில், ஆதரிக்கப்படும் சங்கிலியில் குறிப்பிடப்பட்ட பெறுநர் பணப்பைக்கு டொமைன் டோக்கனை Namefi உருவாக்குகிறது. ஒவ்வொரு வழியிலும் ஒரே இயல்புநிலை பணப்பையோ சங்கிலியோ இருக்கும் என்று கருதாமல், நீங்கள் பயன்படுத்தும் துல்லியமான பதிவு மற்றும் கட்டண வழிக்கான தற்போதைய ஆவணத்தைச் சரிபாருங்கள். டோக்கன் கட்டுப்பாட்டு அடுக்குடன் கூடுதலாக ரெஜிஸ்ட்ரார், பதிவகம், கொள்கை, ஒப்பந்தம், தகராறு மற்றும் சட்ட அடுக்குகள் தொடர்ந்தும் உள்ளன.

### நான் விரும்பும் சரியான பெயர் ஏற்கெனவே எடுக்கப்பட்டிருந்தால் என்ன செய்வது?
அதற்குத்தான் மேலே கூறிய பல பெயர் கிடைப்புத்தன்மைச் சரிபார்ப்பு — ஒன்றன்பின் ஒன்றாகச் சோதிப்பதற்குப் பதிலாக, பல பெயர்களை ([TLD](/ta/glossary/tld/) மாற்றுகள், முன்னொட்டுகள், இணைச்சொற்கள்) முகவருக்கு வழங்கி, உண்மையில் எவை கிடைக்கின்றன என்பதைத் தெரிவிக்கச் சொல்லுங்கள்.

### இதை முயற்சிப்பதற்கு முன்பு எனக்கு Namefi கணக்கு தேவையா?
கிடைப்புத்தன்மையைச் சரிபார்க்க நிதி தேவையில்லை; இணக்கமான MCP பயன்பாடு, நிலையான API key-க்கு பதிலாக OAuth மூலம் அங்கீகரிக்க முடியும். அடிப்படை availability REST செயல்பாடு பொதுவானதாக ஆவணப்படுத்தப்பட்டிருந்தாலும், நேரடி MCP endpoint `401 Unauthorized` எனப் பதிலளித்தது; அங்கீகரிக்கப்படாத `initialize` கோரிக்கைக்கான அந்தப் பதில் ஜூலை 14, 2026 அன்று சரிபார்க்கப்பட்டது.

## நீங்கள் ஏற்கெனவே இருக்கும் பணியோட்டத்திலேயே பெயரையும் வெளியிடுங்கள்

ஹோஸ்டிங் தளத்தைத் தேர்ந்தெடுப்பதுபோல டொமைனும் ஓர் உள்கட்டமைப்புத் தீர்மானம். அங்கீகாரத்திற்காக OAuth ஒருமுறை உலாவியைத் திறக்கலாம்; அதற்குப் பிறகு பதிவு மற்றும் DNS பதிவு பணியோட்டம் கோடிங் முகவர் உரையாடலுக்குள்ளேயே தொடரலாம். அடுத்த முறை ஒரு முகவர் தளத் துணைடொமைனில் இயங்கும் செயலியை உங்களிடம் அளிக்கும்போது, ஒரு பெயரைச் சரிபார்க்கச் சொல்லுங்கள்.

இணக்கமான பயன்பாட்டில் OAuth-ஐப் பயன்படுத்துங்கள் அல்லது **[Namefi API key ஒன்றை உருவாக்குங்கள்](https://namefi.io/api-key)**; பின்னர் இப்போது நீங்கள் உருவாக்கிக்கொண்டிருப்பதில் அதை முயற்சியுங்கள். முழு வழிகாட்டிக்கு [Namefi MCP Quickstart: Claude Code, Cursor & Windsurf](/ta/blog/mcp-quickstart/) கட்டுரையைப் பாருங்கள்.
