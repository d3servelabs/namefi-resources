예전에는 “도메인 하나 사 줘”라고 하면 브라우저를 열고 검색창에 이름을 입력한 다음, 개인정보 보호와 이메일 호스팅 같은 추가 상품을 권하는 페이지를 지나 카드 번호를 입력하는 과정을 뜻했습니다. 2026년에는 점점 더 많은 구매자에게 이 말이 채팅창에 문장 하나를 입력하고 나머지 과정이 진행되는 것을 지켜본다는 뜻이 되었습니다. 이것이 사람들이 말하는 “자연어 도메인 구매”입니다. 다만 이 표현은 꽤 느슨하게 쓰이므로, 실제로 무엇이 필요한지 정확히 짚어 볼 필요가 있습니다.

이 가이드는 하나의 전체 사례를 대화 순서대로 살펴봅니다. 한쪽에는 사람이 일상적인 언어로 요청한 내용이 있고, 다른 쪽에는 [AI 에이전트](/ko/glossary/ai-agent/)가 실제로 수행하는 작업이 있습니다. 그리고 대부분의 가이드가 건너뛰는 부분, 즉 에이전트가 사용자의 말을 API에 그대로 전달하는 데 그치지 않고 판단해야 하는 지점도 설명합니다. 여기서는 [Namefi](https://namefi.io)를 구체적인 사례로 사용하지만, 프롬프트에서 실제 도메인 등록까지 이어지는 방식은 특정 업체만의 기능이 아닙니다. 글의 후반부에서는 이 점도 솔직하게 비교합니다.

## “자연어 구매”가 실제로 의미하는 것

서로 매우 다른 두 가지 방식이 모두 “AI로 도메인 구매하기”라고 불립니다. 이 둘을 혼동하는 데서 대부분의 오해가 시작됩니다.

첫 번째는 **채팅 인터페이스를 씌운 이름 생성기**입니다. 사업을 설명하면 도구가 등록 가능한 이름을 제안하고, 그중 하나를 클릭하면 일반적인 등록대행자 결제 페이지로 이동합니다. 장바구니도, 계정 생성도, 수동으로 검색할 때 보게 되는 “연 $9.99에 개인정보 보호 추가” 같은 추가 판매도 그대로입니다. AI는 아이디어를 구상하는 단계를 줄였을 뿐, 구매 과정 자체를 줄이지는 않았습니다.

두 번째는 에이전트가 **대화의 일부로 구매를 실행하는 방식**입니다. 채팅을 떠나지 않은 채 등록 가능 여부를 확인하고, 계정 잔액을 기준으로 실제 가격을 알려 주며, 사용자가 확인하면 도메인을 등록하고 DNS까지 설정합니다. 이를 위해서는 에이전트가 단순히 문장을 생성하는 것이 아니라 실제로 호출할 수 있는 API가 있어야 합니다. 사용자가 대화 중인 클라이언트가 [Model Context Protocol](https://modelcontextprotocol.io) (MCP) 서버에 연결되어 있거나, 대화 도중 호출할 수 있는 도메인 등록대행자의 실제 작업을 도구로 노출하는 일반 REST API에 연결된 스크립트여야 합니다.

구분하는 방법은 간단합니다. AI가 확인 가능한 주문과 함께 도메인이 *등록되었다고* 알려 주나요? 아니면 “결제 계속하기” 버튼까지만 안내하나요?

## 시작하기 전에 필요한 것

두 가지가 필요합니다. 도메인 등록대행자의 API에 연결된 에이전트와 결제 수단입니다. Namefi에서는 Claude, Cursor, Windsurf 등 여러 MCP 지원 클라이언트를 Namefi MCP 서버에 연결한 뒤 API 키 또는 암호화폐 [지갑](/ko/glossary/wallet/)으로 인증하면 됩니다. 정확한 설정은 클라이언트마다 다르므로 이 글의 범위를 벗어납니다. [AI 에이전트로 Namefi에서 도메인을 등록하는 방법](/ko/blog/ai-agent-register/)은 표준 설정 가이드이고, [Claude로 도메인 구매하기: Namefi MCP 단계별 가이드](/ko/blog/claude-mcp-domains/)는 Claude에 맞춰 명령어 하나하나를 설명한 버전입니다. 이 글은 해당 설정이 이미 끝났다고 가정하고, 그다음 단계인 실제 대화부터 시작합니다.

## 전체 대화 단계별 살펴보기

다음은 주석을 곁들인 하나의 전체 실행 사례입니다. 인용문은 사용자가 입력할 법한 내용이고, 해설은 내부에서 어떤 일이 일어나는지, 그리고 에이전트가 요청을 그대로 전달하지 않고 판단해야 하는 지점을 설명합니다.

**1. 양식의 필드가 아니라 원하는 결과를 설명합니다.**

> “Aurorafield라는 새 프로젝트에 사용할 도메인이 필요해. 먼저 aurorafield.com을 확인하고, 이미 등록되어 있으면 비슷한 대안을 몇 개 제안해 줘.”

이 요청은 단순히 “aurorafield.com을 검색해 줘”가 아닙니다. 대안까지 포함된 목표입니다. 에이전트는 이를 실제로 확인할 수 있는 작업으로 나눠야 합니다. 먼저 정확한 이름의 등록 가능 여부를 조회하고, 이미 등록된 경우에 대비해 후보들을 일괄 조회할 준비를 합니다. 글자 그대로만 처리하는 에이전트는 조회할 때마다 요청을 다시 말해 달라고 합니다. 유용한 에이전트는 사용자가 의도한 순서를 추론합니다.

**2. 에이전트가 확인하고, 단순한 예 또는 아니요 이상의 정보를 알려 줍니다.**

> “aurorafield.com은 등록할 수 있습니다. 다른 후보는 확인할 필요가 없었습니다.”

내부에서는 도메인 등록 가능 여부를 확인합니다. 인증이나 지출 승인이 필요 없는 읽기 전용 조회이므로, 사용자가 구매를 결정하기 전에 실행해도 안전합니다. 신중한 에이전트라면 *무엇을* 확인했는지도 함께 알려 줍니다. 그래야 비용이 발생하기 전에 잘못 들었거나 오타가 난 이름을 발견할 수 있습니다.

**3. 사용자가 확인합니다. 전체 흐름에서 의도적으로 내려야 하는 유일한 구매 결정입니다.**

> “응, 1년 동안 등록해 줘. 활성화되면 루트 도메인은 203.0.113.10을 가리키게 하고, `www`도 같은 곳을 가리키도록 리디렉션을 추가해 줘.”

이 한 메시지는 세 가지 일을 합니다. 지출을 승인하고, 등록 기간을 정하며, DNS를 지정합니다. 좋은 에이전트는 “비용이 들고 조용히 되돌릴 수 없는 작업”과 “단순한 설정 작업”을 구분합니다. 사용자의 잔액을 기준으로 정확한 비용을 다시 알려 줌으로써 “응, 등록해 줘”가 무조건적인 승인이 아니라 충분한 정보를 바탕으로 한 확인이 되게 합니다.

**4. 에이전트가 등록을 제출하고, 즉시 끝난 것처럼 가장하지 않습니다.**

> “등록을 제출했습니다. 비동기로 처리되므로 완료되었다고 가정하지 않고 끝날 때까지 상태를 확인하겠습니다. 잠시만 기다려 주세요.”

도메인 등록은 한 번의 요청과 응답으로 끝나지 않습니다. 레지스트리가 처리해야 하는 주문입니다. 에이전트는 등록을 제출한 뒤 주문이 최종 상태인 `SUCCEEDED`, `FAILED`, `CANCELLED`, `PARTIALLY_COMPLETED` 중 하나에 도달할 때까지 상태를 조회합니다. 요청을 제출하자마자 성공했다고 말하는 에이전트는 실제로 *일어난 일*이 아니라 자신이 *요청한 일*을 설명하는 것입니다. 정직한 에이전트는 도메인이 사용자의 것이 되었다고 말하기 전에 최종 상태를 기다립니다.

**5. 주문이 성공하면 에이전트가 DNS를 설정하고, 별도 지시 없이도 형식을 정확히 맞춥니다.**

> “aurorafield.com이 등록되었습니다. 루트에 A 레코드를 설정해 203.0.113.10을 가리키게 하고, `www`도 같은 곳을 가리키도록 CNAME을 설정했습니다. 두 레코드 모두 활성화되었습니다.”

이 단계에는 사용자가 신경 쓰지 않도록 자연어 계층이 대신 처리해야 할 형식상의 함정이 있습니다. DNS 레코드의 `rdata` 값이 호스트 이름이면 끝에 점이 필요합니다. 예를 들어 CNAME 대상은 `aurorafield.com.`이어야 합니다. 반면 영역 이름 자체에는 끝에 점이 없어야 합니다. 이 둘을 반대로 처리하면 DNS 쓰기 요청이 거부되는 경우가 많습니다. 사용자는 끝에 점을 입력하지 않았습니다. “www를 같은 곳으로 연결해 줘”라는 말을 올바른 형식의 두 레코드 유형으로 변환하는 것이 바로 대화형 방식이 사용자를 대신해 처리해야 할 판단입니다.

**6. 이후 요청도 평범한 말로 이어서 하면 됩니다.**

> “실수로 도메인을 잃지 않도록 자동 갱신도 켜 줘.”

새로 설정하거나 배울 도구가 없습니다. 같은 대화가 계속됩니다. 이것이 진정한 장점입니다. 각 단계를 수동으로 처리할 수 없어서가 아니라, 확인, 가격 조회, 승인, 등록, 대기, 설정, 조정이 여섯 개의 별도 화면이 아니라 하나의 대화에서 이루어지기 때문입니다.

과정이 끝나면 실제 [ICANN](/ko/glossary/icann/) 공인 등록과 요청한 DNS 설정을 갖게 됩니다. 또한 Namefi에서는 기본적으로 데이터베이스의 한 행에만 기록되는 것이 아니라 지갑에 보관되는 NFT 형태의 [토큰화 도메인](/ko/glossary/tokenized-domain/)을 받습니다. 이 모든 과정에 결제 페이지는 필요하지 않습니다.

## 사용자가 계속 관여해야 하는 지점

이 대화를 읽으면 사람은 첫 메시지를 입력하고 마지막 메시지를 확인하기만 하면 된다고 생각하기 쉽습니다. 하지만 그것은 잘못된 결론입니다.

도메인을 등록할 수 있는 에이전트는 실제 돈을 지출하고, 이미 실사용 트래픽을 처리하는 대상의 DNS를 다시 설정할 수도 있습니다. 위 대화가 매끄럽게 진행된 이유는 구매 전에 정확히 한 번, 3단계에서 확인이 이루어졌고 그 전후의 모든 작업은 비용이 들지 않거나 사용자가 명시적으로 요청했기 때문입니다. 이는 우연이 아니라 사용자가 의도적으로 정해야 할 정책입니다.

- **명시적인 확인이 필요한 작업을 정하세요.** 등록 가능 여부 확인 같은 읽기 전용 조회는 위험이 없으므로 확인할 필요가 없습니다. 작업이 돈을 지출하거나 이미 활성화된 대상을 변경하는 순간부터는 “먼저 물어보기”가 원칙이어야 합니다.
- **대화를 시작하기 전에 에이전트의 지출 한도를 정하세요.** Namefi에서는 API 키에 연결된 잔액에 얼마를 충전할지 정하면 됩니다. 감독 없이 에이전트가 사용해도 괜찮은 금액만 충전하세요.
- **자격 증명의 범위를 좁게 제한하세요.** 새로 등록한 도메인을 소유할 지갑으로만 제한하고, 대화 도중 노출되어서는 안 되는 자산이 들어 있는 지갑은 사용하지 마세요.
- **승인하기 전에 DNS 변경 내용을 검토하세요.** 다른 인프라 변경을 검토할 때와 마찬가지입니다. 에이전트가 위에서 설명한 끝점 규칙처럼 *구문*을 정확히 처리하더라도, 사용자가 말한 “같은 곳”을 잘못 이해하면 레코드를 엉뚱한 대상으로 연결할 수 있습니다.

[에이전트 네이티브 도메인 등록대행자란 무엇인가?](/ko/blog/agent-native/)에서는 모든 등록대행자의 에이전트용 인터페이스에 적용할 수 있는 일반적인 체크리스트를 더 자세히 설명합니다. [AI 에이전트로 Namefi에서 도메인을 등록하는 방법](/ko/blog/ai-agent-register/)의 가드레일 섹션에서는 Namefi 설정에 맞춰 같은 내용을 다룹니다.

## Cloudflare와 Name.com에서의 같은 접근 방식

이 방향으로 나아가는 등록대행자는 Namefi만이 아닙니다. 2026년 4월부터 베타로 제공되는 Cloudflare Registrar API는 [AI 에이전트가 브라우저 조작이나 수동 승인 없이 도메인의 등록 가능 여부를 검색하고, 가격을 확인하며, 프로그래밍 방식으로 등록을 완료할 수 있게 합니다](https://webhosting.today/2026/04/22/ai-agents-can-now-register-domains-no-human-required/#:~:text=lets%20an%20AI%20agent%20search%20for%20domain%20availability%2C%20check%20pricing%2C%20and%20complete%20registration%20programmatically%20without%20any%20browser%20interaction%20or%20manual%20approval). 업체의 API만 다를 뿐, 위 사례와 매우 비슷한 형태의 대화입니다. Name.com도 같은 변화를 겨냥한 유사한 “AI 네이티브” 포지셔닝을 중심으로 API를 새로 구축했습니다.

어떤 등록대행자를 사용하든 위의 가드레일은 중요하므로 한계도 솔직하게 짚어야 합니다. Cloudflare 베타를 다룬 업계 기사에서는 [베타 발표에 에이전트별 지출 한도나 등록 승인 워크플로가 설명되어 있지 않다](https://webhosting.today/2026/04/22/ai-agents-can-now-register-domains-no-human-required/#:~:text=The%20beta%20announcement%20does%20not%20describe%20per-agent%20spending%20limits%20or%20registration%20approval%20workflows)고 명확히 지적했습니다. 위에서 말한 “시작하기 전에 결정하라”는 조언이 내장 기능이 아니라 부족한 부분으로 드러난 사례입니다. 또한 다른 곳에서는 여전히 구매가 아니라 제안에 그치는 방식이 흔합니다. 예를 들어 Wix는 웹사이트 빌더 안에서 AI가 이름을 제안하는 방법을 다룬 자체 가이드 [AI를 활용해 도메인 이름을 구매하는 방법](https://www.wix.com/blog/buy-a-domain-name-with-ai)을 제공합니다. 이는 이 글의 도입부에서 두 번째 방식과 구분한 첫 번째 유형의 “AI 도메인 구매”입니다.

각 에이전트 네이티브 등록대행자가 가격, 결제, DNS 관리, 토큰화 소유권 측면에서 실제로 무엇을 지원하는지 자세히 알아보려면 [Cloudflare vs Name.com vs Namefi: AI 에이전트 네이티브 등록대행자 비교](/ko/blog/cf-namecom-namefi/)를 참고하세요.

## 자주 묻는 질문

### 도메인 이름을 제안하는 챗봇과 정말 다른가요?

네. 차이는 제안이 아니라 구매에 있습니다. 이름을 제안하는 챗봇은 “등록 가능한 이름은 다음과 같습니다. 하나를 클릭해 결제를 진행하세요”에서 끝납니다. 자연어 구매 흐름은 대화를 떠나지 않은 채 상태를 확인할 수 있는 주문과 등록 완료된 도메인까지 이어집니다.

### 에이전트가 제게 묻지 않고 돈을 지출할 수도 있나요?

위의 권장 방식대로 설정했다면 그러면 안 됩니다. 읽기 전용 조회는 비용이 들지 않고 확인도 필요하지 않습니다. 잔액을 사용하는 모든 작업은 명시적인 동의를 기다리도록 설정해야 합니다. 이는 기술 자체에 내재된 특성이 아니라 사용자가 정하는 정책입니다.

### 에이전트에게 정확한 도메인 이름을 주지 않으면 어떻게 되나요?

유능한 에이전트는 “내 커피숍에 쓸 이름인데 가능하면 짧은 것으로” 같은 모호한 요청을 먼저 검색하고 제안하는 단계로 처리합니다. 실제 구매는 사용자가 구체적인 이름을 확인한 뒤에만 진행됩니다.

### 등록 주문을 넣은 뒤 취소할 수 있나요?

주문이 성공 최종 상태에 도달하면 다른 실제 도메인과 같습니다. 일반적인 등록대행자의 취소 및 환불 정책이 적용되며, 에이전트를 사용했다는 이유로 특별한 “실행 취소” 기능이 생기지는 않습니다. 그렇기 때문에 등록 전 확인 단계가 대화의 어느 지점보다 중요합니다.

### 이런 방식으로 등록하면 도메인이 자동으로 토큰화되나요?

Namefi에서는 기본적으로 그렇습니다. 다른 지갑을 지정하지 않으면 새로 등록한 도메인이 API 키에 연결된 지갑으로 Base상의 NFT로 발행되어, 표준 ICANN 등록과 함께 이전 가능한 온체인 소유권을 제공합니다. 자세한 내용은 [토큰화 도메인이란 무엇인가?](/ko/glossary/tokenized-domain/)를 참고하세요.

### 이런 방식으로 대화하려면 Namefi API를 배워야 하나요?

아니요. 그것이 바로 이 방식의 핵심입니다. 위 대화의 모든 과정은 일상적인 문장으로 이루어집니다. API와 정확한 요청 형식은 에이전트가 호출하기 위해 내부에 존재할 뿐, 사용자가 읽기 위한 것이 아닙니다. 작동 방식을 직접 확인하려면 [Claude로 도메인 구매하기: Namefi MCP 단계별 가이드](/ko/blog/claude-mcp-domains/)에서 각 단계의 기반 작업 이름과 함께 같은 흐름을 살펴볼 수 있습니다.
